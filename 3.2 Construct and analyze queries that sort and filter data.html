SQL Master Guide: Concepts & Logic

A guide designed to clarify the "Why" behind the commands.

1. The Grammar of Sorting

ORDER BY

The specific command used to sort data.

Common Confusion: ORDER BY vs. SORT BY

The Logic: Think of SQL as a strict English teacher.

  • SORT BY - This is the action you want to take, but it is not the command.
  • ORDER BY - This is the only valid grammar. Using "Sort By" will cause an error.

2. Filtering: WHERE vs. HAVING

The "When Does It Happen?" Test

The difference isn't just about what they filter, but when they filter.

Command What it Filters When it Runs The "Math" Test
WHERE Raw Rows (Individual records) Before Grouping/Math "Is the Date in the last 7 days?" (No math needed)
HAVING Groups/Aggregates (Totals) After Grouping/Math "Is the Total Sales > $10,000?" (Math required)
Why use WHERE for Dates?

You asked: "Why WHERE and not HAVING for dates?"

Logic: You want to throw away the "bad dates" before you waste time doing math on them. HAVING would force the computer to calculate everything first, then throw it away (inefficient).

3. Aggregation (Grouping)

The "Aggregate Anchor" Rule

If you use a function like SUM, COUNT, or MAX, the database collapses many rows into one.

Why do I need GROUP BY if I'm not filtering totals?

You asked: "Why is there GROUP BY even though there's no HAVING?"

Logic: Because you used SUM(Amount). You cannot display a "Total Sum" (1 number) next to "Salesperson" (100 names) unless you tell the database to Group the names into piles first.

4. Subquery Logic: ALL vs. ANY

The "Zero Results" Trap

Comparing a single ID to a list of IDs.

Keyword Logic Translation Outcome
= ALL "I must be Person A AND Person B at the same time." 0 Results (Impossible Logic)
= ANY "I must be Person A OR Person B." Success (Returns matching records)
Clarification on "3 Variables"

You asked: "It doesn't say 'match everyone' in the text?"

Logic: The word ALL implicitly creates an "AND" chain.
ID = 101 AND ID = 102. Since one number cannot be two different numbers simultaneously, the result is always zero.

Syntax Rule: No Double WHERE

Wrong: WHERE ID = WHERE...

Right: WHERE ID = ANY (SELECT...)

Logic: WHERE opens the door. Inside, you must use operators (=, >, ANY), not another door opener.

5. Handling NULL Values

IS NOT NULL
Why not "<> NULL"?

The Ghost Analogy: NULL is not a value (like 0 or "Zero"). It is a ghost (empty space).

  • <> NULL: "Is this number different from a ghost?" -> The computer gets confused (Unknown).
  • IS NOT NULL: "Is there something real in this box?" -> Correct.

6. String Concatenation

The "Space" Spacer

SELECT Address1 + ' ' + Address2

Why the quotes ' '?

You asked: "Why are there quotes around the empty space?"

Logic: The + sign is super-glue. If you don't glue a spacer in the middle, words smash together.

  • Without quotes: 3 Main StApt 4 (Unreadable)
  • With quotes: 3 Main St Apt 4 (Readable)

Since a "space" is text, SQL requires single quotes around it.
